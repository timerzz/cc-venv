# ccv list 设计文档

## 1. 命令定位

`ccv list` 的作用是：

- 列出所有命名环境
- 展示每个环境的基础摘要
- 作为 CLI 和 Web 列表页的共同数据模型参考

它的目标是“快速浏览现有环境”，不是做完整扫描或健康检查。

---

## 2. 数据来源

`list` 面向目录：

```text
~/.ccv/envs/
```

第一版数据来源分两层：

1. 目录枚举
   - 扫描 `~/.ccv/envs/` 下一级子目录
   - 每个一级子目录视为一个候选环境
2. 摘要读取
   - 优先读取 `<env>/ccv.json`
   - 不默认触发实时完整扫描

设计原则：

- 环境目录存在，优先视为环境存在
- `ccv.json` 用于补充摘要，不作为存在性的唯一判据

---

## 3. 输出目标

第一版 `list` 至少应输出：

- 环境名
- 环境根路径
- 摘要信息

摘要信息建议围绕 `ccv.json` 中已存在的数据，保持轻量，例如：

- 是否存在 `CLAUDE.md`
- `skills` 数量
- `agents` 数量
- `commands` 数量
- `rules` 数量
- `plugins` 数量
- `mcpServers` 数量

如果摘要缺失，第一版允许退化为：

- 只显示名称和路径

---

## 4. 排序与展示

第一版建议：

- 默认按环境名升序输出

CLI 展示原则：

- 可读优先
- 不展开资源明细列表
- 不在 `list` 中展示过多诊断信息

Web 列表页也应复用同样的排序和摘要口径。

---

## 5. 错误与降级策略

`list` 应尽量稳，不因单个环境摘要异常而整体失败。

建议策略：

- `~/.ccv/envs/` 不存在
  - 返回空列表
- 某个环境缺少 `ccv.json`
  - 仍列出该环境，摘要留空
- 某个环境的 `ccv.json` 解析失败
  - 仍列出该环境，并标记为摘要不可读

设计原则：

- `list` 是浏览命令，不是校验命令
- 单个坏环境不应阻断全局查看

---

## 6. 与扫描的关系

`list` 第一版默认不做实时完整扫描。

原因：

- 环境数量多时，实时扫描成本高
- `list` 的目标是快速展示，不是生成精确导出清单

因此：

- `export` 前必须扫描
- `list` 默认读取已有摘要

如果后续需要更强一致性，可以再引入：

- `ccv list --refresh`

但这不属于第一版必需能力。

---

## 7. 与 Web 的关系

`ccv web` 的环境列表页应尽量复用 `list` 的数据模型：

- 同样的环境枚举方式
- 同样的默认摘要来源
- 同样的排序规则

这样可以避免 CLI 和 Web 列表语义漂移。

---

## 8. 实现边界

`list` 的实现边界建议保持简单：

- `internal/env`
  - 负责枚举环境目录
- `internal/config`
  - 负责读取 `ccv.json` 摘要
- `internal/cli`
  - 负责格式化输出

不在 `list` 中做：

- 完整资源扫描
- 导出校验
- 环境修复
- 交互式确认
